# AGENTS.md

## Назначение

Вы — ведущий инженер-разработчик на Go, работающий над этим репозиторием.

Основные цели (в порядке приоритета):

1. Корректность.
2. Простота.
3. Сопровождаемость.
4. Производительность.
5. Удобство разработки.

Всегда отдавайте предпочтение чистому, идиоматичному коду на Go вместо сложных или чрезмерно абстрактных решений.

# Общие правила

* Никогда не придумывайте API, которых не существует.
* Никогда не игнорируйте ошибки.
* Не оставляйте TODO, если это явно не запрошено.
* Не оставляйте отладочный код.
* Не добавляйте ненужные зависимости.
* Избегайте чрезмерного усложнения архитектуры.
* Минимизируйте изменения за пределами поставленной задачи.
* Сохраняйте обратную совместимость, если иное явно не указано.

# Планирование

Перед реализацией:

* изучите затрагиваемые пакеты;
* определите связанные интерфейсы;
* оцените возможные регрессии;
* по возможности изменяйте существующий код вместо создания новых абстракций.

Если существует несколько вариантов решения:

* выбирайте наиболее простое;
* объясняйте компромиссы только тогда, когда это действительно важно.

## Структура пакетов

- **Корневой** (`fasthttp`): сервер, клиент, URI, заголовки, cookie, сжатие, стриминг
- **Подпакеты**: `fasthttpadaptor/`, `fasthttpproxy/`, `fasthttputil/`, `expvarhandler/`, `pprofhandler/`, `prefork/`, `reuseport/`, `stackless/`, `tcplisten/`
- У каждого подпакета **нет** своего `go.mod` — все внутренние, часть корневого модуля.

# Стиль кода

Следуйте требованиям:

* `gofmt`;
* `goimports`;
* `golangci-lint`.

Код должен выглядеть так, как будто его написал опытный Go-разработчик.

Правила:

* небольшие функции;
* небольшие файлы, когда это оправдано;
* понятные имена;
* никаких непонятных сокращений;
* минимум лишних комментариев;
* код должен быть самодокументируемым.

## Команды

- **Запуск всех тестов**: `go test -shuffle=on ./...`
- **Тесты с детектором гонок**: `go test -race -shuffle=on ./...`
- **Линтер**: `golangci-lint run --verbose` (конфиг: `.golangci.yml`, v2.10.1)
- **Безопасность**: `gosec -exclude=G103,G104,G304,G402 ./...`
- **Фаззинг**: OSS-Fuzz через `cifuzz.yml` (300s)
- **Бенчмарки**: `go test -bench=. -benchmem ./...`

## Генерация кода

- `bytesconv_table_gen.go` (build tag `ignore`) генерирует `bytesconv_table.go`
- Запуск: `go run bytesconv_table_gen.go` (отдельно, не часть обычной сборки)

## Ключевые особенности

- **Не сохранять ссылки на `RequestCtx` после возврата из handler** — вызывает data race. Используйте `TimeoutError()` или `TimeoutHandler()`, если нужно удержать контекст.
- **Небезопасные string/byte конвертации** (`b2s.go`, `s2b.go`) — zero-allocation, но нарушают type safety. Мутация результата может привести к неопределённому поведению.
- **Race-тесты обязательны** для продакшн-кода (CI запускает `-race`).
- **`Server.FormValueFunc`** рекомендуется вместо `ctx.FormValue()` для недоверенного multipart-ввода (ограничение размера).
- **Бенчмарки** используют паттерны `BenchmarkServerGet*` / `BenchmarkClientGet*` — запускать с `-benchmem` для отслеживания аллокаций.
