# План: Client disconnect during streaming

## 1. Описание сценария

Клиент разрывает соединение с прокси во время стриминга — либо при передаче тела запроса (POST/PUT), либо при получении тела ответа (GET). Прокси должен корректно завершить соединение с upstream: не возвращать повреждённое соединение в пул и не тратить ресурсы на чтение данных, которые уже никому не нужны.

Сценарий распадается на **два подтипа**:

| Подтип                                         | Момент разрыва                                     | Фаза в `handle()`                             | Риск                                                                                        |
|------------------------------------------------|----------------------------------------------------|-----------------------------------------------|---------------------------------------------------------------------------------------------|
| **A. Client disconnect during request body**   | Клиент обрывает POST-запрос во время передачи тела | `writeRequestBody()` → `PipeCopy()`           | Upstream получает неполное тело, прокси возвращает 502                                      |
| **B. Client disconnect during response body**  | Клиент обрывает соединение после начала ответа     | `streamResponseBody()` → fasthttp пишет ответ | Upstream продолжает слать данные, соединение может быть возвращено в пул с неполным чтением |

---

## 2. Подтип A: Client disconnect during request body

### Что происходит в коде

`handler.writeRequestBody()` (handler.go:152–186) вызывает `pool.PipeCopy(h.ctx.Request.BodyStream(), h.connection)`. `h.ctx.Request.BodyStream()` — это стрим, который fasthttp читает из клиентского соединения.

Если клиент обрывает соединение:
1. `BodyStream()` возвращает ошибку при очередном `Read()`
2. `PipeCopy()` возвращает эту ошибку
3. `writeRequestBody()` возвращает `false`
4. `handle()` вызывает `pool.CloseAndDropUpstreamConnection()`
5. На `ctx` устанавливается **502 Bad Gateway**

**Проблема:** upstream уже получил часть тела запроса. Он может начать обрабатывать неполный запрос. Прокси не отправляет upstream уведомление об обрыве (RST), просто закрывает соединение.

### Существующее покрытие

Уже есть тест `TestWriteRequestBodyStreamPipeCopyError` — он использует `errWriter` на upstream conn, что симулирует ошибку записи в upstream. Но нет теста, который симулирует **ошибку чтения из клиентского стрима** (client body stream → error).

### Реализация

**Mock:** `errReader` для `ctx.Request.BodyStream()`:

```go
// Используем существующий partialReader или errReader как BodyStream.
// При PipeCopy src.Read() вернёт ошибку → PipeCopy вернёт ошибку → 502.
```

**Unit-тест:** проверить, что при ошибке чтения из клиентского body stream `writeRequestBody` возвращает `false` и upstream соединение закрывается.

**Интеграция:** реальный клиент, который обрывает POST-запрос после отправки части тела.

### Проверка

- `writeRequestBody()` возвращает `false`
- `pool.CloseAndDropUpstreamConnection()` вызван
- `ctx.Response.StatusCode() == 502`

## 3. Подтип B: Client disconnect during response body

### Что происходит в коде

После `streamResponseBody()` (handler.go:223–235) `handle()` возвращает управление fasthttp. fasthttp начинает писать ответ клиенту:

1. Заголовки уже отправлены (`ImmediateHeaderFlush = true`)
2. fasthttp читает тело из `PoolReader` (который читает из `bufIOReader` → upstream `net.Conn`)
3. fasthttp пишет прочитанные данные в клиентское соединение

Если клиент обрывает соединение во время шага 3:
- fasthttp получает ошибку записи в клиентский сокет
- fasthttp прекращает чтение из `PoolReader`
- `PoolReader` **продолжает читать из upstream**, пока не получит EOF или лимит байт

**Две проблемы:**

1. **Wasted upstream bandwidth:** прокси продолжает читать данные, которые уже никому не нужны
2. **Повреждённое соединение в пул (для Content-Length):** если `PoolReader.readWithLimit()` дочитает до `remain=0`, соединение будет возвращено в пул как живое, хотя upstream мог отправить данные, которые никто не прочитал. Следующий запрос получит «хвост» предыдущего ответа.

### Текущее поведение

| Content-Length  | Поведение PoolReader                                      | Риск                                                                          |
|-----------------|-----------------------------------------------------------|-------------------------------------------------------------------------------|
| `>= 0`          | `readWithLimit()` дочитает до `remain`, вернёт conn в пул | **Да:** conn возвращён как живой, но в буфере могут быть непрочитанные данные |
| `< 0` (chunked) | `readUntilEOF()` получит EOF, закроет conn                | **Нет:** conn закрыт, пул не получает повреждённое соединение                 |

### Требуемое поведение (ideal)

Прокси должен **прекратить чтение из upstream** при обрыве клиента и **закрыть upstream соединение** (не возвращать в пул). Для этого нужно:

1. После `streamResponseBody()` не возвращаться из `handle()` сразу, а контролировать процесс стриминга
2. Или использовать `ctx.BodyStreamWriter()` вместо `ctx.SetBodyStream()`, чтобы иметь контроль над стримингом
3. Или проверять `ctx.IsClientDisconnect()` перед/во время чтения из upstream

Но **текущая архитектура** не позволяет этого без изменений в `handle()`. `streamResponseBody()` только устанавливает стрим и возвращается. Контроль над чтением тела находится в fasthttp.

### Тестирование текущего поведения

**Цель:** проверить, что при обрыве клиента во время стриминга ответа:
- `handle()` не падает
- Upstream соединение корректно закрывается (не зависает)
- Для Content-Length: соединение возвращается в пул (текущее поведение, документируем как known issue)

## Критерии приёмки

| # | Критерий                                                                         | Подтип | Проверка                                |
|---|----------------------------------------------------------------------------------|--------|-----------------------------------------|
| 1 | Ошибка чтения клиентского body stream → 502                                      | A      | `StatusCode() == 502`                   |
| 2 | При ошибке body stream upstream соединение закрыто                               | A      | `CloseAndDropUpstreamConnection` вызван |
| 3 | Client disconnect во время Content-Length стриминга: соединение возвращено в пул | B1     | Соединение не висит, пул не блокирован  |
| 4 | Client disconnect во время chunked стриминга: соединение закрыто                 | B2     | `CloseAndDropUpstreamConnection` вызван |
| 5 | Все тесты проходят с `-race`                                                     | все    | `go test -race` без data race           |

## Различия с upstream disconnect (предыдущий сценарий)

| Аспект              | Upstream disconnect                            | Client disconnect                                            |
|---------------------|------------------------------------------------|--------------------------------------------------------------|
| Читатель            | Прокси читает из upstream → ошибка             | Прокси пишет клиенту → ошибка                                |
| Фаза в handle()     | `readResponseHeaders` или `streamResponseBody` | `writeRequestBody` (A) или после `handle()` (B)              |
| Статус клиенту      | 502 (всегда)                                   | 502 (A) / 200 + неполное тело (B)                            |
| Upstream conn       | Закрывается (CloseAndDrop)                     | Закрывается (A) / возвращается в пул (B1) / закрывается (B2) |
| Тест через handle() | Да, статус проверяется сразу                   | Только A; для B требуется чтение тела                        |
